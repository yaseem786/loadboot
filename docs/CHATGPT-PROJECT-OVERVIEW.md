## 1. WHAT WE ARE BUILDING
**LoadBoot** is a US truck-dispatch platform: shippers post freight → licensed freight brokers handle the
transaction → LoadBoot's staff **Command Center** + matching engine route each load to the best carrier →
carriers haul it, tracked live, paid through recorded bank-rail settlements. Revenue: a flat dispatch fee
(currently 5%) charged to carriers. Ambition: Uber Freight / Amazon Relay depth, designed for
hundreds-of-millions-of-accounts scale.

**Two founding principles enforced in code:**
1. **100% automation** — every data point flows automatically to whoever needs it. Humans exist ONLY at:
   money movement, verification/violations, and legal sign-off.
2. **No fake data** — every number on every screen is computed from real database rows and states its basis.

## 2. TECH STACK & ARCHITECTURE
- **Marketing website:** static HTML generated by a Python builder (`build_site.py` + `motifs_module.py`),
  published by Netlify from branch `preview/command-center-v1` (build: `python3 build_site.py`, publish `site/`).
- **App portals (5), all vanilla-JS ES modules, no framework:**
  - `/app/carrier` — the carrier's whole business (the flagship product)
  - `/app/partner` — brokers (transactional) + shippers/facilities (request scope)
  - `/app/command-center` — staff ops (60+ screens)
  - `/app/developer` — API keys/docs portal
  - `/app/pocket` — driver PWA companion
  All server calls go through wrappers in `app/shared/api.js`.
- **Backend:** Supabase Postgres ×2 — STAGING `snslhvmkjusozgjelghi`, PROD `rwscphuhpjoudvljvmdk`.
  ALL business logic lives in Postgres `SECURITY DEFINER` RPC functions (`cc_*`), ~360+ of them.
  Data tables live in a private schema (`app_private`) — no direct table access from clients.
- **Tenancy/authorization:** `my_carrier_org()` / `my_partner_org(kind)` resolve the caller's org from JWT;
  staff actions gated by `has_global_permission('<key>')` (keys like dispatch.manage, finance.approve,
  comm.send, analytics.view, settings.manage).

### Security invariants (held through the ENTIRE build, re-verified after every migration)
- The anonymous (logged-out) role can execute **exactly 5** SECURITY DEFINER functions — never more.
- Every migration applied to BOTH databases with **md5 function-body parity** proven.
- Every increment proven by a **live SQL security matrix**: a DO block simulates each identity
  (staff / carrier / broker / shipper / outsider / anon) via the JWT GUC and asserts every allow AND every
  deny; seeded rows self-clean. ~45 matrices run this build, **all PASS**.
- Automation runners are `service_role`-only; production feature flags (`auto_dispatch`,
  `referral_program`) are **OFF in prod** until the owner flips them (staging ON, fully tested).

## 3. THE AUTOMATED VALUE CHAIN (end-to-end, all live & matrix-proven)
1. **Shipper posts a load tender** (`cc_shipper_post_load`) — industry-REQUIRED fields enforced server-side:
   ref/PO, pickup+delivery contacts, commodity/weight/pieces, equipment, cargo value, facility notes,
   dock hours, appointment flag. Hazmat without UN/placard info = hard legal STOP. Incomplete tenders are
   rejected with the missing fields NAMED.
2. **Open broker pool** — every active licensed broker instantly sees the full decision detail;
   first-come **Claim** (concurrency-safe) or Command-Center assignment.
3. **Broker quotes → tenders to dispatch** — becomes a dispatch load through a DB trigger **ready-gate**
   that FORCES the complete industry rate card before any load can surface:
   detention $/hr + free hours, layover/day, TONU, lumper policy, plus FCFS/appointment scheduling.
   Defaults come from a versioned, staff-editable **rate-standards library** (detention $60/hr after 2h free,
   layover $250/day, TONU $250, lumper reimbursed-with-receipt, etc.). Every post records a **rate agreement**
   row (who agreed, when, exact snapshot, standards version). Carriers later see the same card VERBATIM.
4. **Matching engine** — hard eligibility filter then explainable ranking (the score always equals the sum
   of displayed factors — no black box). **Auto-dispatch runner** (`cc_auto_dispatch_run`, service-role,
   flag-gated): auto-offers every ready load to the top-ranked carriers, re-reading each carrier's LIVE
   dispatch preferences (min $/mile, equipment, lanes) AND live service listings (hazmat, team, liftgate,
   tarp, TWIC, white-glove, drop-and-hook, expedited) on EVERY run. Proven live: a $5/mi min-rate carrier is
   never offered a $2.57/mi load; listing "Hazmat" unblocks hazmat loads on the very next run.
5. **Carrier accepts** (recorded written approval) → transactional booking (single winner, stale-version
   guard, competing offers expired, trip + document checklist created) → a **Rate Confirmation
   auto-issues** — an immutable 18-item snapshot (no UPDATE path exists in the schema), carrier acknowledges
   once → a **30-item Dispatch Sheet** is assembled live for portal + driver app.
6. **In transit:** consent-first GPS tracking (mandatory from book to delivery, consent recorded once),
   arrive/depart stamps that measure real detention minutes against the agreed free time, proof-required
   emergency/reschedule flow with predefined categories, staff exception center + Trip Control Tower.
7. **Delivery:** POD upload → staff review → **Delivery Doc Pack** (POD/RC/invoice REQUIRED; dwell stamps
   and lumper receipts CONDITIONAL; `packet_ready` flag; 3-year retention rule) → invoice → finance
   lifecycle (receivables, payables, reconciliation; maker-checker on all settlements).
8. **Referral program, 100% automated:** `?ref=` links auto-captured on the website → auto-claimed at first
   portal entry → commissions AUTO-ACCRUE via DB trigger the moment a fee-bearing invoice is sent/paid
   (multi-level chain, hold period, auto-promotion to payable) → referrer requests payout with bank details →
   the ONLY human gate: staff approve/mark-paid. Influencers/affiliates get a full earnings dashboard
   without needing a trucking company.
9. **Payments rail (bank-account default):** every org manages its own bank profile (masked to last-4 even
   for the owner, staff-verified, any edit resets verification). **No code path moves money automatically,
   anywhere** — all transfers are recorded with maker-checker approval.
10. **Account Health (Amazon-Seller-Central style), every account type:** carrier, broker AND shipper each
    scored 0–100 with role-specific industry duties; score always = 100 − itemized deductions (insurance
    expiring, docs missing, tracking gaps, detention abuse, quote SLA misses, tender completeness…);
    tiers healthy ≥90 / at-risk 70–89 / critical <70; staff violations ledger (warning −5, violation −15,
    critical −40; written reason REQUIRED; instant notification to the account); Command Center board sorts
    worst-first with inline issue/resolve. No manual score editing exists — corruption-proof by design.
11. **Pre-book compliance gate:** one go/no-go RPC consolidating the dispatcher's final checks + 19 STOP/
    REJECT conditions (authority inactive, insurance lapsed/insufficient for cargo value, packet incomplete,
    equipment mismatch, weight overage, health critical…). HOS legal duty explicitly remains with
    driver+carrier, stated on the sheet.

## 4. INDUSTRY COMPLIANCE SOP (the legal/documentation backbone)
The owner's complete 3-level US-trucking documentation spec is canon at `docs/INDUSTRY-COMPLIANCE-SOP.md`,
with every item tagged [LEGAL] / [REQUIRED] / [CONDITIONAL] / [OPTIONAL]. Implemented as D1–D8, ALL SHIPPED:
- **D1** Load-tender completeness (per-load required fields, hazmat legal stop)
- **D2** Onboarding packet engine — 30 seeded templates: broker (MC authority, $75k BMC-84/85 bond [LEGAL],
  W-9, contact sheet…), carrier (authority, auto liability [LEGAL], cargo insurance, W-9, SAFER check,
  ELD confirmation…), shipper (facility profile, credit app, dock policies…). Submit → staff verify /
  reject / waive lifecycle, written reasons required, Command Center review board.
- **D3** Master agreements engine — versioned Broker↔Shipper (30 clauses) / Broker↔Carrier (16) /
  Dispatcher↔Carrier (30) skeletons; **publication is impossible without explicit legal-counsel
  confirmation** (a boolean the staff must assert); acceptance recorded once per org per version.
- **D4** Immutable Rate Confirmations (18 items, trigger-issued at booking, acknowledge-once)
- **D5** 30-item Dispatch Sheets (portal modal + driver-app inline)
- **D6+D7** Pre-book gate + 19-condition STOP/REJECT engine
- **D8** Delivery doc pack + 3-year records retention rule
UI everywhere: packet checklist cards (carrier Account tab + broker portal), agreement-accept card,
RC/Sheet/DocPack modals on trips, driver-app inline sheet, CC onboarding board inside Account Health.
